# 스위치온 다이어트 앱 아키텍처 (Switch-on Diet App Architecture)

본 문서는 스위치온 다이어트 앱의 프론트엔드 아키텍처 및 폴더 구조를 설명합니다.

## 1. 아키텍처 개요 (Architecture Overview)

본 프로젝트는 **기능 중심 아키텍처 (Feature-Based Architecture)**를 채택하여 설계되었습니다. 
각 도메인(기능)별로 코드를 응집시켜 유지보수성과 확장성을 높였습니다.

### 핵심 원칙
1. **관심사 분리 (Separation of Concerns):** 각 기능(Feature)은 독립적으로 동작할 수 있도록 구성되며, 다른 기능과의 결합도를 최소화합니다.
2. **도메인 중심 (Domain-Driven):** `home`, `calendar`, `diet`, `chat`, `my`, `history` 등 실제 사용자가 인지하는 비즈니스 도메인을 기준으로 폴더를 분리합니다.
3. **재사용성 (Reusability):** 여러 기능에서 공통으로 사용되는 UI 컴포넌트나 유틸리티는 최상위 `components` 또는 `utils` 폴더에 위치시킵니다.
4. **실제 데이터 기반 UI (Real-Data Driven UI):** 캘린더, 히스토리 등의 UI는 실제 날짜(예: 시작일 기준)와 일차(1일차, 2일차)를 정확히 매핑하여 렌더링합니다.

## 2. 폴더 구조 (Directory Structure)

```text
src/
├── App.tsx                  # 앱의 최상위 진입점 및 라우팅/전역 상태 관리
├── main.tsx                 # React DOM 렌더링
├── index.css                # 글로벌 스타일 (Tailwind CSS)
│
├── features/                # 도메인/기능별 모듈 (Feature Modules)
│   ├── home/                # 홈 화면 (기상, 수면 기록, 오늘의 요약)
│   │   └── HomeScreen.tsx
│   ├── calendar/            # 캘린더 화면 (월간/주간 진행률, 일별 상태)
│   │   └── CalendarScreen.tsx
│   ├── diet/                # 식단 관련 기능 (식단 가이드, 기록, 히스토리)
│   │   ├── DietScreen.tsx
│   │   ├── RecordMealScreen.tsx
│   │   └── DietHistoryScreen.tsx
│   ├── chat/                # AI 코치 채팅 기능
│   │   └── ChatScreen.tsx
│   ├── history/             # AI 코치 피드백 히스토리 (일간/주간 총평)
│   │   └── HistoryScreen.tsx
│   └── my/                  # 마이페이지 (설정, 내 정보)
│       └── MyScreen.tsx
│
└── components/              # 공통 UI 컴포넌트
    ├── ui/                  # 버튼, 모달, 카드 등 재사용 가능한 기본 UI 컴포넌트
    └── layout/              # 네비게이션 바(BottomNav), 헤더 등 레이아웃 컴포넌트
```

## 3. 기능별 모듈 상세 (Feature Modules Detail)

### 3.1. `home` (홈)
* **역할:** 앱 실행 시 가장 먼저 보이는 화면으로, 사용자의 현재 상태(기상 여부, 수면 시간, 오늘 먹어야 할 식단 요약)를 보여줍니다.
* **주요 로직:** 기상 시간 체크, 취침 시간 누락 확인, AI 코치 브리핑.

### 3.2. `calendar` (캘린더)
* **역할:** 28일간의 다이어트 여정을 시각적으로 보여줍니다.
* **주요 로직:** 
  * 사용자의 실제 다이어트 시작일(예: 2026.03.13)을 기준으로 요일과 날짜를 정확히 매핑합니다.
  * 각 날짜 셀에 'N일차' 정보와 실제 '일(Date)' 정보를 함께 표시합니다.
  * 날짜별 상태(완벽 성공, 주의 필요, 미입력) 계산 및 렌더링.

### 3.3. `diet` (식단)
* **역할:** 1~4주차 전체 식단 가이드라인 제공, 실제 식단 기록, 과거 식단 기록 조회.
* **주요 컴포넌트:**
  * `DietScreen`: 1주차(장 비우기)부터 4주차(유지기)까지 전체 4주 플랜의 허용/금지 식품 및 식단표 가이드라인. 현재 진행 중인 주차를 강조 표시합니다.
  * `RecordMealScreen`: 텍스트 기반의 식단 입력 폼.
  * `DietHistoryScreen`: 타임라인 형태의 과거 식단 기록 및 스킵/위반 여부 피드백.

### 3.4. `chat` (채팅)
* **역할:** AI 코치와의 1:1 대화 인터페이스.
* **주요 로직:** 사용자의 질문에 대한 다이어트 전문 답변 제공.

### 3.5. `history` (AI 코치 피드백 히스토리)
* **역할:** 사용자의 식단 및 수면 기록을 바탕으로 AI가 분석한 데일리 총평 및 주간 피드백을 모아보는 화면.
* **주요 로직:** 
  * 'N일차 데일리 총평'과 'N주차 총평'을 구분하여 표시합니다.
  * 각 피드백은 날짜, 핵심 주제, 태그(#식단완벽, #수면주의 등)를 포함하여 한눈에 파악하기 쉽게 UI를 구성합니다.

### 3.6. `my` (마이페이지)
* **역할:** 사용자 프로필, 설정, 각종 히스토리(식단, AI 피드백)로 이동하는 네비게이션 허브.

## 4. 향후 리팩토링 계획 (Future Refactoring Plans)

1. **UI 컴포넌트 분리:** `HomeScreen` 등에서 반복적으로 사용되는 모달(Modal), 다이얼로그(Dialog), 카드(Card) UI를 `src/components/ui`로 추출.
2. **상태 관리 분리:** 컴포넌트 내부에 있는 복잡한 상태 로직(예: 시간 계산, 상태 분기)을 Custom Hook(`useWakeTime`, `useDietHistory` 등)으로 분리하여 `features/{feature}/hooks/`에 위치.
3. **상수 및 유틸리티 분리:** 하드코딩된 데이터(예: 모크 데이터, 날짜 포맷팅 함수)를 `utils` 또는 `constants` 디렉토리로 분리.
4. **전역 상태 관리 도입:** 다이어트 시작일, 현재 일차, 사용자 프로필 등의 전역 데이터를 관리하기 위해 Context API 또는 Zustand 도입 고려.
